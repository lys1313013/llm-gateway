# LLM Gateway 需求文档

> **版本**：v1.0 · **状态**：已交付 · **最后更新**：2026-05-26

---

## 1. 项目概述

LLM Gateway 是一个大模型 API 兼容网关，同时兼容 **OpenAI** 和 **Anthropic** 两种 API 协议。服务将客户端请求透明转发到第三方兼容 API（如阿里云 DashScope、AWS Bedrock 等），并提供管理后台用于配置管理、请求日志查看和 Token 用量统计。所有配置和日志数据持久化到 PostgreSQL 数据库。

### 1.1 目标用户

| 角色 | 诉求 |
|------|------|
| 应用开发者 | 用统一的 Base URL 接入多种大模型，开发阶段不依赖真实 API Key |
| 运维 / SRE | 集中管理模型路由、监控 Token 消耗、排查请求日志 |
| QA / 测试 | 验证客户端对 OpenAI / Anthropic 协议的兼容性 |

### 1.2 系统架构

## 2. 功能需求

### 2.1 API 接口

#### 2.1.1 OpenAI 兼容接口

| 接口 | 说明 |
|------|------|
| `POST /v1/chat/completions` | 聊天补全，支持流式（SSE）和非流式响应 |
| `GET /v1/models` | 返回可暴露的模型列表 |

- 支持工具调用（`tool_calls`）和函数调用（`function_call`）透传
- 流式响应支持 `reasoning_content`（推理内容）字段聚合

#### 2.1.2 Anthropic 兼容接口

| 接口 | 说明 |
|------|------|
| `POST /v1/messages` | Messages API，支持流式（SSE）和非流式响应 |

**请求兼容字段**：`model`、`messages`、`system`、`max_tokens`、`stop_sequences`、`temperature`、`top_p`、`top_k`、`metadata`、`stream`、`tools`、`thinking`

**消息格式**：
- `messages` 支持纯文本和 Content Block 数组，角色支持 `user` / `assistant`
- `system` 支持字符串和 Content Block 数组
- `tools` 使用 Anthropic 格式（`name`、`description`、`input_schema`）
- 图片输入支持 `base64` 编码（`image/jpeg`、`image/png`、`image/gif`、`image/webp`）

**非流式响应**：返回 `id`（`msg_` 前缀）、`type: "message"`、`role`、`content` 数组（`text` / `tool_use`）、`model`、`stop_reason`、`usage`

**流式 SSE 事件**：

| 事件 | 携带内容 |
|------|---------|
| `message_start` | 完整 `message` 对象（含初始 `usage`） |
| `content_block_start` | `index` + `content_block` 对象 |
| `content_block_delta` | `index` + `delta`（文本：`{type, text}`；思考：`{type, thinking}`） |
| `content_block_stop` | `index` |
| `message_delta` | `stop_reason` + 更新后的 `usage` |
| `message_stop` | 流结束信号 |
| `ping` | 保活 |

**错误响应**：`{type: "error", error: {type, message}}`，HTTP 状态码与原错误保持一致（400/401/403/404/429/500）

**特有请求头**：`x-api-key`（认证）、`anthropic-version`（版本）、`anthropic-beta`（实验特性）、`anthropic-dangerous-direct-browser-access`

---

### 2.2 代理转发

网关将请求透明转发到配置的第三方 API，不缓存、不修改响应内容。

| 项 | OpenAI | Anthropic |
|----|--------|-----------|
| 认证方式 | `Authorization: Bearer <api_key>` | `x-api-key` + `anthropic-version` |
| 转发地址 | `base_url` + `/chat/completions` | `base_url` + `/v1/messages` |
| 流式处理 | SSE 分块透传，后台聚合记录日志 | SSE 事件透传，后台聚合记录日志 |
| 错误格式 | OpenAI 错误格式 | Anthropic 错误格式 |

**通用行为**：
- 转发前可选覆盖模型名（由路由规则的 `target_model` 配置）
- 请求超时返回 `504`，连接失败返回 `502`
- 超时时间按路由规则配置，默认 60 秒

---

### 2.3 模型路由

路由系统决定每个请求转发到哪个上游厂商。

**核心规则**：
- 按 `priority` 降序 + `id` 升序遍历所有启用规则，`fnmatch` 通配符匹配模型名
- 一条路由天然覆盖目标厂商已配置的所有协议（OpenAI / Anthropic 各自独立判断）
- 入站请求的协议由 URL 路径决定：`/v1/chat/completions` → openai，`/v1/messages` → anthropic
- 命中首条即停止，无匹配则返回 `404`
- 规则存储在数据库中，变更即时生效，无需重启

**路由规则字段**：

| 字段 | 说明 |
|------|------|
| `model_pattern` | 通配符（如 `gpt-4*`、`claude-*`、`*`） |
| `provider_id` | 目标厂商 |
| `target_model` | 目标模型名（可选，不填保留原始名） |
| `timeout` | 超时秒数，默认 60 |
| `log_requests` / `log_responses` | 日志开关 |
| `priority` | 优先级，数值越大越先匹配 |
| `is_active` | 启用开关 |

---

### 2.4 请求日志

- 所有 API 请求和响应全量记录到 `api_logs` 表
- 记录字段：时间、协议、模型、是否流式、状态码、处理耗时（ms）、Token 用量、目标地址、请求体（JSONB）、响应体（JSONB）、错误信息
- 流式请求在传输完成后聚合记录

**查询支持**：

| 参数 | 说明 |
|------|------|
| `limit` / `offset` | 分页 |
| `model` | 按模型名模糊搜索（`ILIKE`） |
| `protocol` | 按协议过滤（`openai` / `anthropic`） |

---

### 2.5 Token 用量统计

**估算策略**：

| 场景 | 方式 |
|------|------|
| 上游返回 `usage` | 直接使用 |
| OpenAI 协议，无 usage | `tiktoken` 按模型选编码，未知模型回退 `cl100k_base` |
| Anthropic 协议，无 usage | `cl100k_base` 近似估算，分别计算 input/output |
| 估算失败 | 回退到字符长度 / 3 |

统一存储为 `prompt_tokens` / `completion_tokens` / `total_tokens`。

**统计接口**：

| 接口 | 说明 |
|------|------|
| `/api/logs/today_stats` | 当日请求数 + Token 汇总 |
| `/api/stats/daily_tokens` | 日期范围查询：多日返回每日汇总，单日返回每小时明细（24 小时，按 `DB_TIMEZONE`） |

**管理后台可视化**：
- 统计卡片：总请求数、总 Token、Input Token、Output Token
- 每日 / 每小时 Token 消耗趋势图（堆叠柱状图）
- 模型 Token 消耗占比饼图
- Token 用量明细表

---

### 2.6 管理后台

基于 React + Ant Design 的 Web UI，通过 RESTful API（`/api` 前缀）管理配置。

**大模型厂商（Provider）**：管理第三方 API 厂商连接信息。每个厂商可独立配置 OpenAI / Anthropic 两种协议的 Base URL（同一个厂商不同协议的接口地址往往不同），API Key 厂商共用一份。同一厂商只需配置一次，路由也只需配置一次即可覆盖两种协议。

**模型路由（Model Route）**：管理路由规则，路由不再区分协议。字段参见 [2.3 模型路由](#23-模型路由)。

**暴露模型（Exposed Model）**：管理 `GET /v1/models` 返回的模型列表，支持对每个模型进行 OpenAI / Anthropic 协议连通性测试，记录最近测试成功时间。

## 3. 数据库设计

### 3.1 api_logs（请求日志表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `created_at` / `updated_at` | TIMESTAMP WITH TIME ZONE | 时间戳 |
| `model` | VARCHAR(100) | 模型名 |
| `protocol` | VARCHAR(50) | 协议（`openai` / `anthropic`） |
| `is_stream` | BOOLEAN | 是否流式 |
| `status_code` | INTEGER | HTTP 状态码 |
| `processing_time_ms` | INTEGER | 处理耗时（ms） |
| `prompt_tokens` / `completion_tokens` / `total_tokens` | INTEGER | Token 用量 |
| `target_url` | VARCHAR(255) | 转发目标地址 |
| `request_data` / `response_data` | JSONB | 请求体 / 响应体 |
| `error_message` | TEXT | 错误信息 |

### 3.2 provider（厂商表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `name` | VARCHAR(100) UNIQUE | 厂商名称 |
| `openai_base_url` | VARCHAR(255) | OpenAI 协议接口地址（可空） |
| `anthropic_base_url` | VARCHAR(255) | Anthropic 协议接口地址（可空） |
| `api_key` | VARCHAR(255) | API 密钥（两种协议共用） |
| `remark` | TEXT | 备注 |
| `create_time` / `update_time` | TIMESTAMP WITH TIME ZONE | 时间戳 |

**约束**：`openai_base_url` 与 `anthropic_base_url` 至少需要填写一个（保证厂商至少能被一种协议路由到）；`api_key` 必须填写。

### 3.3 model_route（模型路由表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `model_pattern` | VARCHAR(255) | 通配符匹配规则 |
| `route_type` | VARCHAR(20) | 路由类型（`proxy`） |
| `provider_id` | INTEGER (FK → provider) | 目标厂商 |
| `target_model` | VARCHAR(100) | 目标模型名（可选） |
| `timeout` | INTEGER | 超时秒数，默认 60 |
| `log_requests` / `log_responses` | BOOLEAN | 日志开关 |
| `priority` | INTEGER | 优先级，默认 0 |
| `is_active` | BOOLEAN | 启用开关 |
| `create_time` / `update_time` | TIMESTAMP WITH TIME ZONE | 时间戳 |

### 3.4 exposed_model（暴露模型表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `model_id` | VARCHAR(255) UNIQUE | 模型 ID |
| `owned_by` | VARCHAR(100) | 所属组织 |
| `is_active` | BOOLEAN | 启用开关 |
| `last_openai_test_time` | TIMESTAMP WITH TIME ZONE | 最近 OpenAI 协议测试时间 |
| `last_anthropic_test_time` | TIMESTAMP WITH TIME ZONE | 最近 Anthropic 协议测试时间 |
| `create_time` / `update_time` | TIMESTAMP WITH TIME ZONE | 时间戳 |

---

## 4. 非功能需求

- **动态配置**：所有配置通过管理后台或 Admin API 动态修改，存储在数据库，无需重启服务
- **持久化**：配置和日志均持久化到 PostgreSQL
- **前后端分离**：前端与后端可独立部署，开发时通过 Vite 代理（`/api`、`/v1`）转发到后端
- **跨域支持**：内置 CORS，允许前端 Web 或跨端应用直接调用

## 5. 管理后台 API 汇总

| 模块 | 资源路径 | 方法 |
|------|----------|------|
| 厂商 | `/api/provider[/<id>]` | GET / POST / PUT / DELETE |
| 路由 | `/api/route[/<id>]` | GET / POST / PUT / DELETE |
| 模型 | `/api/exposed_model[/<id>]` | GET / POST / PUT / DELETE |
| 模型测试 | `/api/exposed_model/<id>/test_time` | PUT |
| 请求日志 | `/api/logs` | GET（`limit` / `offset` / `model` / `protocol`） |
| 今日统计 | `/api/logs/today_stats` | GET |
| Token 统计 | `/api/stats/daily_tokens` | GET（`start_date` / `end_date`） |
