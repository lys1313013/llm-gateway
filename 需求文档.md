# Mock OpenAI 需求文档

## 1. 概述

Mock OpenAI 是一个大模型 API 兼容代理服务，同时兼容 OpenAI 和 Anthropic 两种 API 协议。服务将客户端请求透明转发到第三方兼容 API（如阿里云 DashScope、AWS Bedrock 等），并提供管理后台用于配置管理、请求日志查看和 Token 用量统计。所有配置和日志数据持久化存储到 PostgreSQL 数据库。

---

## 2. 功能需求

### 2.1 OpenAI 兼容接口

- 提供与 OpenAI `/v1/chat/completions` 兼容的聊天补全接口
- 提供与 OpenAI `/v1/models` 兼容的模型列表接口
- 支持流式（SSE）和非流式响应
- 支持工具调用（tool_calls）和函数调用（function_call）的透传
- 流式响应支持 `reasoning_content`（推理内容）字段的聚合

### 2.2 Anthropic 兼容接口

- 提供与 Anthropic `/v1/messages` 兼容的 Messages API 接口
- 支持流式（SSE）和非流式响应

**请求格式兼容：**
- 支持 `model`、`messages`、`system`、`max_tokens`、`stop_sequences`、`temperature`、`top_p`、`top_k`、`metadata`、`stream` 等标准字段
- `messages` 支持 `user` 和 `assistant` 角色，支持纯文本和 Content Block 数组两种 content 格式
- `system` 支持字符串和 Content Block 数组两种格式
- `tools` 支持 Anthropic 工具定义格式（`name`、`description`、`input_schema`）
- 支持 `thinking`（扩展思考）参数
- 支持图片输入（`base64` 编码，`image/jpeg`、`image/png`、`image/gif`、`image/webp` 格式）

**响应格式兼容：**
- 非流式响应返回 `id`（`msg_` 前缀）、`type: "message"`、`role: "assistant"`、`content` 数组、`model`、`stop_reason`、`stop_sequence`、`usage`
- `content` 数组元素包含 `type`（`text` 或 `tool_use`）、`text`（文本内容）或 `name`/`id`/`input`（工具调用参数）
- `usage` 格式为 `{input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens}`
- 文本 content block 包含服务端生成的 `id`（`msg_` 前缀）

**流式响应（SSE）兼容：**
- SSE 事件类型：`message_start`、`content_block_start`、`content_block_delta`、`content_block_stop`、`message_delta`、`message_stop`、`ping`
- `message_start` 携带完整的 `message` 对象（包含 `id`、`type`、`role`、`model`、`usage` 初始值等）
- `content_block_start` 携带 `index` 和完整的 `content_block` 对象
- `content_block_delta` 携带 `index` 和 `delta` 对象（文本类型为 `{type, text}`，思考类型为 `{type, thinking}`）
- `content_block_stop` 携带 `index`
- `message_delta` 携带 `delta`（`stop_reason`、`stop_sequence`）和更新后的 `usage`
- `message_stop` 为流结束信号
- 定期发送 `ping` 事件保持连接

**错误响应格式：**
- 返回 Anthropic 格式的错误响应：`{type: "error", error: {type: "error_type", message: "description"}}`
- HTTP 状态码保持一致（400/401/403/404/429/500 等）

**Anthropic 特有的请求头处理：**
- 支持 `x-api-key` 认证头（替代 OpenAI 的 Authorization Bearer）
- 支持 `anthropic-version` 版本控制头
- 支持 `anthropic-beta` 实验特性头
- 支持 `anthropic-dangerous-direct-browser-access` 头

### 2.3 代理模式（上游转发）

- 将请求透明转发到第三方兼容 API
- **OpenAI 协议转发**：
  - 使用 `Authorization: Bearer <api_key>` 认证头
  - 转发目标为厂商 `base_url` 拼接 `/chat/completions`
  - 支持流式 SSE 透传，流式分块在后台聚合记录到日志
  - 上游返回错误时，返回 OpenAI 格式的错误响应
- **Anthropic 协议转发**：
  - 使用 `x-api-key` 认证头和 `anthropic-version` 版本头
  - 转发目标为厂商 `base_url` 拼接 `/v1/messages`
  - 支持 Anthropic SSE 事件透传，流式事件在后台聚合记录到日志
  - 上游返回错误时，返回 Anthropic 格式的错误响应
- 转发前可选覆盖请求中的模型名称（通过路由规则中的 `target_model` 配置）
- 支持配置请求超时时间
- 超时返回 504 错误，连接失败返回 502 错误

### 2.4 模型路由

- 根据请求中的模型名称自动选择目标厂商和转发配置
- 自动识别请求协议类型（OpenAI 或 Anthropic），路由规则按协议隔离：OpenAI 请求只匹配 `protocol=openai` 的路由，Anthropic 请求只匹配 `protocol=anthropic` 的路由
- 路由规则支持通配符匹配（使用 `fnmatch`），如 `gpt-4*` 匹配所有 GPT-4 系列模型，`claude-*` 匹配所有 Claude 系列模型，`*` 匹配所有模型
- 路由规则支持优先级排序（`priority` 字段，数值越大越优先），按优先级降序、ID 升序排列，命中第一个匹配的规则后停止
- 每条路由可单独控制是否记录请求日志和响应日志
- 路由规则存储在数据库中，变更无需重启服务
- 未匹配到任何路由规则时，返回 404 错误

### 2.5 请求日志

- 记录所有 API 请求和响应到数据库（`api_logs` 表）
- 日志内容包括：请求时间、协议类型（OpenAI/Anthropic）、模型名称、是否流式、状态码、处理耗时（毫秒）、Token 用量（prompt_tokens/completion_tokens/total_tokens）、目标地址、请求体（JSONB）、响应体（JSONB）、错误信息
- 流式请求在传输完成后，将分块/事件聚合为完整响应后记录
- 支持分页查看日志列表（`limit`/`offset` 参数）
- 支持按模型名称模糊搜索（`model` 参数，使用 `ILIKE`）
- 支持按协议类型过滤（`protocol` 参数）
- 支持查看单条日志的完整请求和响应 JSON 详情

### 2.6 Token 用量统计

- 代理模式下如上游返回 `usage` 信息则直接使用，否则自动估算
- OpenAI 协议使用 `tiktoken` 按模型名称选择对应编码进行估算，未知模型回退到 `cl100k_base` 编码
- Anthropic 协议使用 `cl100k_base` 编码近似估算，分别计算 `input_tokens`（system + messages + tools）和 `output_tokens`（content blocks）
- Token 估算失败时回退到基于字符长度的粗略估算（长度/3）
- 统一存储为 `prompt_tokens`/`completion_tokens`/`total_tokens` 格式
- 提供以下统计查询接口：
  - **今日统计**（`/api/logs/today_stats`）：当日请求数、Prompt Token、Completion Token、Total Token
  - **每日 Token 统计**（`/api/stats/daily_tokens`）：支持日期范围查询，多日返回每日汇总，单日返回每小时明细（24 小时，按 `DB_TIMEZONE` 时区）
  - **模型 Token 统计**：按模型分组统计 Token 消耗，按 `total_tokens` 降序排列
- 管理后台提供可视化统计仪表板：
  - 统计卡片：总请求数、总 Token、Input Token（Prompt Token）、Output Token（Completion Token）
  - 每日/每小时 Token 消耗趋势图（堆叠柱状图）
  - 按模型划分的 Token 消耗饼图
  - Token 用量明细表

### 2.7 管理后台

管理后台通过 Web 界面（React + Ant Design）和 RESTful API 提供以下配置管理功能：

**大模型厂商管理（Provider）：**
- 新增、编辑、删除 API 厂商配置
- 厂商信息包括：名称（唯一）、接口地址（Base URL）、API 密钥、协议类型（OpenAI / Anthropic）、备注

**模型路由管理（Model Route）：**
- 新增、编辑、删除路由规则
- 路由信息包括：模型匹配规则（通配符）、路由类型、目标厂商、目标模型、超时、优先级、日志开关（请求/响应）、启用开关

**模型列表管理（Exposed Model）：**
- 新增、编辑、删除暴露模型（影响 `GET /v1/models` 接口返回的模型列表）
- 模型信息包括：模型 ID（唯一）、所属组织、启用开关
- 支持对暴露模型进行 OpenAI 和 Anthropic 协议的连通性测试，记录最近测试成功时间

## 3. 数据库设计

### 3.1 api_logs（请求日志表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `created_at` | TIMESTAMP WITH TIME ZONE | 创建时间 |
| `updated_at` | TIMESTAMP WITH TIME ZONE | 更新时间 |
| `model` | VARCHAR(100) | 模型名称 |
| `is_stream` | BOOLEAN | 是否流式请求 |
| `status_code` | INTEGER | HTTP 状态码 |
| `processing_time_ms` | INTEGER | 处理耗时（毫秒） |
| `prompt_tokens` | INTEGER | Prompt Token 数 |
| `completion_tokens` | INTEGER | Completion Token 数 |
| `total_tokens` | INTEGER | 总 Token 数 |
| `target_url` | VARCHAR(255) | 转发目标地址 |
| `request_data` | JSONB | 请求体 |
| `response_data` | JSONB | 响应体 |
| `error_message` | TEXT | 错误信息 |
| `protocol` | VARCHAR(50) | 协议类型（openai/anthropic） |

### 3.2 provider（厂商表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `name` | VARCHAR(100) UNIQUE | 厂商名称 |
| `base_url` | VARCHAR(255) | 接口基础地址 |
| `api_key` | VARCHAR(255) | API 密钥 |
| `protocol` | VARCHAR(50) | 协议类型（openai/anthropic） |
| `remark` | TEXT | 备注 |
| `create_time` | TIMESTAMP WITH TIME ZONE | 创建时间 |
| `update_time` | TIMESTAMP WITH TIME ZONE | 更新时间 |

### 3.3 model_route（模型路由表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `model_pattern` | VARCHAR(255) | 模型匹配规则（通配符） |
| `route_type` | VARCHAR(20) | 路由类型 |
| `provider_id` | INTEGER (FK → provider) | 目标厂商 ID |
| `target_model` | VARCHAR(100) | 目标模型名称 |
| `timeout` | INTEGER | 超时时间（秒），默认 60 |
| `log_requests` | BOOLEAN | 是否记录请求日志 |
| `log_responses` | BOOLEAN | 是否记录响应日志 |
| `priority` | INTEGER | 优先级，默认 0 |
| `is_active` | BOOLEAN | 是否启用 |
| `create_time` | TIMESTAMP WITH TIME ZONE | 创建时间 |
| `update_time` | TIMESTAMP WITH TIME ZONE | 更新时间 |

### 3.4 exposed_model（暴露模型表）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | SERIAL | 主键 |
| `model_id` | VARCHAR(255) UNIQUE | 模型 ID |
| `owned_by` | VARCHAR(100) | 所属组织 |
| `is_active` | BOOLEAN | 是否启用 |
| `last_openai_test_time` | TIMESTAMP WITH TIME ZONE | 最近 OpenAI 协议测试时间 |
| `last_anthropic_test_time` | TIMESTAMP WITH TIME ZONE | 最近 Anthropic 协议测试时间 |
| `create_time` | TIMESTAMP WITH TIME ZONE | 创建时间 |
| `update_time` | TIMESTAMP WITH TIME ZONE | 更新时间 |

---

## 4. 非功能需求

- 所有配置通过管理后台或 Admin API 动态修改，存储在 PostgreSQL 数据库中，无需重启服务
- 所有配置和日志持久化存储到数据库
- 前端与后端分离部署，开发时可独立运行
- 支持从旧版 `config.json` 配置自动迁移到数据库（首次启动时检测并迁移）
- 数据库连接参数通过环境变量配置，提供合理默认值
- 前端开发服务器通过 Vite 代理将 `/api` 和 `/v1` 请求转发到后端
