# OpenAI API 模拟服务 (Mock OpenAI) 需求说明文档

## 1. 产品概述
### 1.1 产品背景
在进行大语言模型（LLM）相关应用的开发和自动化测试时，直接调用真实的 OpenAI API 存在以下痛点：
- **成本问题**：频繁的调试和自动化测试会消耗大量的 API Token，产生高昂的费用。
- **网络与稳定性问题**：真实 API 可能受到网络波动、并发限流等影响，导致测试环境不稳定。
- **结果不确定性**：大模型生成的回复具有随机性，难以覆盖特定的边界条件（如特定的错误码、极端的格式输出等），导致自动化测试难以断言。
- **国内环境限制**：部分国内开发环境无法直接访问官方接口，需要平替方案。

### 1.2 产品定位
本产品旨在提供一个**与 OpenAI API 完全兼容的本地化模拟与代理服务**。它可以作为开发和测试环境中的“API 替身”，实现零成本调试、确定性结果输出以及透明的接口代理转发。

## 2. 目标用户
- **前端/后端研发工程师**：在本地开发阶段，无需依赖真实 API Key 即可完成业务逻辑和界面的开发与调试。
- **测试工程师 (QA)**：用于构建稳定、可重复的自动化测试用例，覆盖各种正常及异常的对话场景。

## 3. 核心应用场景
- **场景一：本地断网/零成本开发**
  开发者在没有网络或不想消耗 Token 的情况下，启动该服务，应用即可获得符合 OpenAI 格式的模拟回复，用于联调 UI 或基础业务逻辑。
- **场景二：自动化测试断言**
  测试用例向该服务发送特定的测试文本（如“你好”），服务根据预设规则固定返回预期的结果，使得测试脚本能够进行稳定的结果断言。
- **场景三：复杂交互模拟（Tool Calling/Function Calling）**
  模拟大模型触发工具调用或函数调用的特殊响应格式，验证业务代码对这些特殊结构的解析和处理能力。
- **场景四：透明代理与日志分析**
  作为中间代理层，将请求转发给兼容 OpenAI 格式的其他第三方大模型厂商（如国内的平替大模型），并在代理层记录完整的请求和响应日志，便于排查问题。

## 4. 功能需求
### 4.1 接口协议兼容
- **标准协议支持**：必须完全兼容 OpenAI `/v1/chat/completions` 接口的请求和响应数据结构。
- **跨域支持**：必须支持跨域资源共享（CORS），允许前端 Web 应用直接在浏览器中向本服务发起调用。

### 4.2 双运行模式支持
系统必须支持在以下两种模式中进行切换：
- **模拟（Mock）模式**：拦截请求，返回本地预设的模拟数据。
- **代理（Proxy）模式**：透传请求，将数据转发至真实的目标服务器并返回结果。

### 4.3 模拟模式（Mock）核心能力
- **条件路由匹配**：支持根据请求体的特征（如模型名称、用户 ID、特定的消息内容等）进行精准匹配。
- **多形态响应支持**：
  - 支持普通文本对话的模拟响应。
  - 支持模拟旧版函数调用（Function Calling）的响应结构。
  - 支持模拟新版工具调用（Tool Calls）的响应结构。
- **规则兜底机制**：当收到的请求未匹配到任何预设规则时，系统需提供一个格式标准的默认响应，确保调用方不会崩溃。
- **动态规则生效**：修改预设的匹配规则和响应内容后，系统应能立即在下一次请求中生效，无需重启服务。

### 4.4 代理模式（Proxy）核心能力
- **请求透传**：支持配置目标服务器地址和 API 鉴权密钥，将请求原样转发。
- **模型重写**：支持在转发过程中，强制替换请求体中的模型名称（例如将请求中的 `gpt-4` 替换为目标厂商对应的模型名称）。
- **日志聚合**：支持在控制台打印完整的请求体和目标服务器返回的响应体，方便开发者追踪调试。

### 4.5 流式输出（Streaming）支持
- 无论是在 Mock 模式还是 Proxy 模式下，当客户端请求参数中指定开启流式输出（`stream: true`）时，系统必须以 Server-Sent Events (SSE) 格式返回数据。
- **拟真打字机效果**：在 Mock 模式下，系统需对预设的完整文本进行分块切割，并模拟真实大模型的输出延迟（逐字或逐块发送），确保前端界面的打字机效果能得到真实还原。
- **流式结构兼容**：流式输出的数据块（Chunks）必须完全符合 OpenAI 的增量输出规范。

## 5. 非功能需求
- **易用性**：配置方式需简单直观，最好通过修改结构化文档（如 JSON）即可完成预设规则的编排。
- **轻量化**：作为本地辅助工具，服务应尽可能轻量，启动速度快，占用系统资源少。
- **稳定性**：在处理高并发的自动化测试请求时，服务不应出现崩溃或内存泄漏；在代理模式下，需具备基础的超时控制和异常处理机制（如目标服务器无响应时返回标准的 504/502 错误结构）。
